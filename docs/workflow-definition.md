# 14ステータスワークフロー定義書

## ステータス一覧

| # | ステータス | フェーズ | 定義 | 責任者 | 次アクション |
|---|-----------|---------|------|--------|-------------|
| 1 | **Icebox** | Planning | 将来やるかもしれないが今は着手しない | PM/PO | 優先度が上がったら「進行待ち」へ |
| 2 | **進行待ち** | Planning | 優先度がついたが未着手 | PM/PO | 要件定義の準備ができたら「要件作成中」へ |
| 3 | **要件作成中** | Planning | 要件定義・仕様書を作成中 | PM/PO | 要件確定後「デザイン待ち」へ（デザイン不要なら「アサイン待ち」へ） |
| 4 | **デザイン待ち** | Design | デザイナーのアサインを待っている | PM | デザイナーが着手したら「デザイン作成中」へ |
| 5 | **デザイン作成中** | Design | UI/UXデザインを作成中 | Designer | デザインレビュー完了後「アサイン待ち」へ |
| 6 | **アサイン待ち** | Dev | 開発者のアサインを待っている | Tech Lead | 開発者を決定したら「開発待ち」へ |
| 7 | **開発待ち** | Dev | アサイン済みだが開発未着手 | Developer | 開発着手時に「開発中」へ |
| 8 | **開発中** | Dev | 実装作業中 | Developer | PR作成時に「コードレビュー」へ |
| 9 | **コードレビュー** | Dev | PRのレビュー待ち/レビュー中 | Reviewer | Approve後「テスト中」へ |
| 10 | **テスト中** | Release | QAテスト実施中 | QA/Developer | テストPass→「リリース待ち」、Fail→「テスト落ち」 |
| 11 | **テスト落ち** | Release | テスト不合格、修正が必要 | Developer | 修正後「開発中」→「コードレビュー」→「テスト中」 |
| 12 | **リリース待ち** | Release | リリース判定待ち | PM/TL | リリース実施後「リリース済み」へ |
| 13 | **リリース済み** | Release | 本番リリース完了 | DevOps | 確認期間後「Done」へ |
| 14 | **Done** | - | 完全完了・クローズ | - | アーカイブ対象 |

## 遷移フロー

```
Icebox → 進行待ち → 要件作成中 → デザイン待ち → デザイン作成中
                                                      ↓
Done ← リリース済み ← リリース待ち ← テスト中 ← コードレビュー ← 開発中 ← 開発待ち ← アサイン待ち
                                       ↑
                                   テスト落ち → 開発中（修正して再テスト）
```

**スキップ**: デザイン不要なIssueは「要件作成中」→「アサイン待ち」へ直接遷移可能

## WIPリミット

| ステータス | リミット | 根拠 |
|-----------|---------|------|
| コードレビュー | 10 | レビュー品質を維持 |
| テスト中 | 5 | QAキャパシティ |
| 開発中 | チーム人数 x 1.5 | マルチタスク防止 |

## 自動化トリガー

| トリガー | アクション |
|---------|-----------|
| Issue/PR作成時 | 「進行待ち」に設定 |
| PR作成時 | 関連Issueを「コードレビュー」へ |
| PRレビュー承認時 | 関連Issueを「テスト中」へ |
| PRマージ時 | 「Done」へ |
| Issueクローズ時 | 「Done」へ |
| Issue再オープン時 | 「進行待ち」へ |
| 同一ステータス5日以上 | Slack/コメントで通知 |
| クローズ後14日 | 自動アーカイブ |
