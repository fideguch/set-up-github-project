# 新ツール導入規則

## 目的

新しい外部 API ツールを my_pm_tools に追加する際の品質ゲート。
「血液が通る状態」（end-to-end 接続成功）を保証するプロセス。

## 導入フロー（MANDATORY）

### Phase 1: API 仕様理解（コード前）

1. **公式ドキュメント読了**: API リファレンス・ガイドを Context7 または WebFetch で取得
2. **認証方式の確認**: OAuth スコープ、API キー、サービスアカウント等
3. **レスポンス型の実物確認**: curl または node で実 API を叩き、レスポンスの形を確認
4. **エッジケースの特定**:
   - 空レスポンス（データなし）の形
   - エラーレスポンスの形（400, 403, 404, 429）
   - 日本語・特殊文字を含むパラメータの扱い
   - ページネーションの有無と形式

### Phase 2: インターフェース設計

1. **型定義**: `src/types/workspace.ts` に readonly 型を追加
2. **GoogleClient 拡張**: インターフェースにメソッド追加
3. **メタデータ取得メソッド**: 動的情報（シート名、フォルダ名等）を事前取得する手段を必ず用意
4. **クエリビルダー**: ユーザー入力を API 仕様に準拠した形に自動変換
   - 例: `searchDrive('test')` → `name contains 'test' and trashed = false`

### Phase 3: 実装

1. **ツール実装**: `src/tools/<category>/<name>.ts`
2. **Zod スキーマ**: `src/schemas/index.ts`
3. **ツール登録**: `src/tools/index.ts` にアノテーション付きで登録
4. **ADC 自動検出**: 認証は環境変数 → ADC → エラーの 3 段階フォールバック

### Phase 4: テスト（3 層）

1. **ユニットテスト**: モック使用、正常系 + エラー系 + エッジケース
2. **モック統合テスト**: `tests/live/` で read→write→rollback サイクル
3. **ライブテスト**: `GOOGLE_LIVE_TEST=1` ゲート付き、実 API 接続確認

### Phase 5: 接続検証（血液検査）

以下が全て成功するまで「導入完了」としない:

- [ ] モックテスト全パス
- [ ] ライブテストで実 API からデータ取得成功
- [ ] 書き込みテストでロールバック成功
- [ ] 日本語パラメータで動作確認
- [ ] 空データ/エラーケースでクラッシュしない

## チェックリスト

新ツール追加 PR では以下を全て満たすこと:

- [ ] 公式ドキュメントの URL を PR 本文に記載
- [ ] レスポンス型が `readonly` で定義されている
- [ ] メタデータ取得メソッドがある（動的情報のハードコード禁止）
- [ ] クエリ/パラメータのバリデーションがある
- [ ] A1 記法等の特殊構文は公式仕様に準拠（単一引用符等）
- [ ] ADC 自動検出に対応
- [ ] 3 層テスト実装済み
- [ ] ライブ接続検証済み（血液検査パス）
